# Revolut plugin for ofxstatement

Converts **Revolut** bank statements to OFX format for import into GnuCash
or other personal finance software.

[ofxstatement](https://github.com/kedder/ofxstatement) is a tool to convert
proprietary bank statements to OFX format.

Two input formats are supported — the format is detected automatically from
the file extension:

| Extension | Format        | How to export                              |
|-----------|---------------|--------------------------------------------|
| `.pdf`    | PDF statement | Revolut app → Statements → Generate        |
| `.csv`    | CSV export    | Revolut app → Statements → Generate        |


## Supported account types

- **Current** (main current account, default)
- **Deposit** (savings / flexible deposit account)
- **Sub-accounts** (e.g. `Andrew`, `MATTHEW` — PDF only, see [Caveats](#caveats))

## Supported currencies

Multi-currency statements are auto-detected:

- **PDF**: currency is read from the PDF header (`USD Statement`, `TRY Statement`, …).
- **CSV**: currency is read from the `Currency` column. Revolut exports one
  CSV per currency account; if a file contains multiple currencies the
  dominant one is used and the rest are dropped.

Currencies with a well-known symbol are written in PDFs as `<symbol><amount>`
(e.g. `$19.61`); others are written as `<amount> <CODE>` (e.g. `29,155.00 TRY`).

| Currency | Format     | Account ID                          |
|----------|------------|-------------------------------------|
| EUR      | `€100.00`  | IBAN                                |
| USD      | `$100.00`  | (empty — set `account_id` manually) |
| GBP      | `£100.00`  | `GB-<sort-code>-<account-number>`   |
| JPY      | `¥100`     | (empty)                             |
| INR      | `₹100.00`  | (empty)                             |
| TRY      | `100.00 TRY` | (empty)                           |
| Other    | `100.00 XXX` (3-letter code) | (empty)           |

For currencies where Revolut doesn't provide a currency-specific account
identifier (no IBAN, no Sort Code), leave `account_id` in configuration
or set it manually per-currency.


### PDF format

The plugin parses the standard PDF account statement exported from the
Revolut app. The PDF must have a text layer (i.e. not a scanned image);
all PDFs generated by Revolut qualify.

The parser handles multi-page statements, multiple date-range sections within
one PDF, continuation/detail lines, multi-currency filtering (only EUR amounts
are extracted), and sub-account sections (pockets and savings).


### Transaction types (PDF)

PDF transactions are classified by matching the description text against
known prefixes. Entries marked ★ are confirmed against real Revolut EUR
statements. Entries marked ○ are best-effort additions — see [Caveats](#caveats).

| Prefix | Description | OFX type | |
|---|---|---|---|
| Transfer to | Outgoing transfer to person/account | XFER | ★ |
| Transfer from | Incoming transfer from person/account | XFER | ★ |
| SWIFT Transfer to | Outgoing SWIFT / international transfer | XFER | ★ |
| SWIFT Transfer from | Incoming SWIFT / international transfer | XFER | ○ |
| Payment from | Incoming payment (SEPA credit transfer) | DEP | ★ |
| Net Interest Paid | Daily interest on savings/deposit | INT | ★ |
| Withheld Tax Refund | Tax refund on interest | INT | ★ |
| Interest earned | Alternate interest label | INT | ○ |
| Premium plan fee | Monthly subscription fee (Premium tier) | FEE | ★ |
| Plus plan fee | Monthly subscription fee (Plus tier) | FEE | ★ |
| Exchanged to | Currency exchange (outgoing leg) | XFER | ★ |
| Exchanged from | Currency exchange (incoming leg) | XFER | ○ |
| To EUR | Move to savings vault / pocket | XFER | ★ |
| From EUR | Move from savings vault / pocket | XFER | ★ |
| To pocket | Move money into a pocket | XFER | ★ |
| Pocket Withdrawal | Withdraw money from a pocket | XFER | ★ |
| Cash withdrawal at | ATM cash withdrawal | ATM | ★ |
| Top-up by | Incoming top-up | DEP | ★ |
| Plan termination refund | Refund of a cancelled plan | FEE | ★ |

Transactions that match none of the above are classified as `DEBIT` or
`CREDIT` based on the amount sign. A word-boundary check for "fee" in
the description is applied before the sign-based fallback.


### CSV format

The CSV export from the Revolut app is a comma-separated file with the
following columns:

| Column | Description |
|---|---|
| Type | Transaction type keyword |
| Product | Account type (`Current`, `Deposit`) |
| Started Date | Initiated timestamp (`YYYY-MM-DD HH:MM:SS`) |
| Completed Date | Completed timestamp — used as the transaction date |
| Description | Free-text description |
| Amount | Transaction amount (positive or negative) |
| Fee | Fee amount (subtracted from the transaction amount) |
| Currency | Currency code (e.g. `EUR`) |
| State | Transaction state (only `COMPLETED` is imported) |
| Balance | Running balance after the transaction |

Fees are subtracted from the transaction amount automatically. Only rows
with `State = COMPLETED` are imported; pending and reverted transactions
are skipped.


### Transaction types (CSV)

CSV transactions are classified by exact match on the `Type` column.
Entries marked ★ are confirmed against real Revolut CSV exports.
Entries marked ○ are best-effort — see [Caveats](#caveats).

| Type | Description | OFX type | |
|---|---|---|---|
| Transfer | SEPA transfers, internal moves, P2P | XFER | ★ |
| Card Payment | Card purchase (VISA/Mastercard) | POS | ★ |
| Topup | Incoming top-up (bank transfer in) | DEP | ★ |
| Exchange | Currency exchange | XFER | ★ |
| Fee | Subscription fee (Plus plan, etc.) | FEE | ★ |
| Charge | Fees (Premium plan, etc.) | FEE | ★ |
| Charge Refund | Refund of a prior charge | FEE | ★ |
| Interest | Interest earned on deposits | INT | ★ |
| ATM | Cash withdrawal | ATM | ★ |
| Card Refund | Card payment refund from merchant | CREDIT | ★ |
| Refund | Other merchant refund | CREDIT | ○ |
| Reward | Cashback or promotional reward | CREDIT | ○ |


### Unused OFX types

The following standard OFX transaction types are not mapped because Revolut
does not distinguish them in its exports:

| OFX type | Meaning | Why not applicable |
|---|---|---|
| `CHECK` | Cheque | Revolut does not issue cheques |
| `CASH` | Cash advance | Not a Revolut product |
| `DIRECTDEP` | Direct deposit (salary) | Incoming credits are not labelled as salary |
| `DIRECTDEBIT` | Direct debit (SEPA mandate) | No SEPA mandate distinction in exports |
| `REPEATPMT` | Standing order | Scheduled payments are not labelled differently |
| `PAYMENT` | Electronic payment | No distinct label in Revolut exports |
| `DIV` | Dividend | Trading statements are separate from bank statements |
| `SRVCHG` | Service charge | Revolut uses "fee" consistently — covered by `FEE` |
| `OTHER` | Catch-all | `DEBIT`/`CREDIT` fallback is more informative |


## Caveats

**CSV exports do not include sub-account data.** The Revolut CSV export
only contains the main account holder's transactions (`Current` and
`Deposit`). Sub-account holders (e.g. family members added via Revolut
\<18 or Revolut Plus/Premium) and their pockets appear only in the PDF
statement. If you need sub-account transactions, use the PDF export.

**Transaction type mappings** marked ○ in the tables above have been added on
a best-effort basis. They have not been verified against real statements.
If you encounter a transaction that is mapped to the wrong OFX type, or one
that falls through to the `DEBIT`/`CREDIT` fallback, please
[open an issue](https://github.com/kedder/ofxstatement-revolut/issues)
and include:

- The transaction type or description prefix
- The OFX type you would expect
- The account type (Current, Deposit, sub-account name)

You do not need to share any amounts, payee names, IBANs, or other personal
information. Enable debug logging (`ofxstatement -d convert …`) to extract
the type safely.

**Secondary-currency amounts are filtered.** Revolut statements often
include both the account's primary currency and a conversion amount in a
second currency (e.g. `€50.00` shown alongside `$52.10` for a currency
exchange). The parser keeps only the primary currency; secondary amounts
are logged at debug level and skipped.


## Installation

Dependencies:

- [ofxstatement](https://github.com/kedder/ofxstatement) — the conversion
  framework this plugin hooks into
- [pdfplumber](https://github.com/jsvine/pdfplumber) — PDF text extraction

Both are declared as package dependencies and installed automatically.

```
pip install ofxstatement-revolut
```

Or from source:

```
git clone https://github.com/kedder/ofxstatement-revolut
cd ofxstatement-revolut
python -m venv .venv
.venv/bin/pip install -e .
```


## Usage

Single file (PDF or CSV — detected by extension):

```
ofxstatement convert -t revolut statement.pdf statement.ofx
ofxstatement convert -t revolut statement.csv statement.ofx
```

Multiple files:

```bash
for f in *.pdf *.csv; do
    ofxstatement convert -t revolut "$f" "${f%.*}.ofx"
done
```

The output file uses your IBAN as the account ID (extracted from the PDF
header, or set manually via configuration), so GnuCash will automatically
associate it with the correct account on re-import.


## Configuration

Configure the plugin via `ofxstatement edit-config`:

```ini
[revolut]
plugin = revolut
account = Current
currency = EUR
account_id = DE92100101787623104264
```

| Option | Default | Description |
|---|---|---|
| `account` | `Current` | Account section to import: `Current`, `Deposit`, or a sub-account owner name (e.g. `Andrew`). Case-insensitive. |
| `currency` | `EUR` | Statement currency (auto-detected from PDF). |
| `account_id` | *(empty)* | Account identifier (e.g. IBAN). Auto-detected from the PDF header if not set. |

Sub-account selection (`account = Andrew`) returns all of that person's
transactions across their current account, pockets, and savings sections.
This only works with PDF statements — see [Caveats](#caveats).

Enable debug logging to see exactly how the parser processes each line:

```
ofxstatement -d convert -t revolut statement.pdf statement.ofx
```
